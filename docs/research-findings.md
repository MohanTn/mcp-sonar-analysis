# Market & Technical Landscape Report: mcp-sonar-analysis

**Date of research:** 2026-06-14
**Scope:** MCP server giving Claude Code live awareness of SonarSource-family static analysis findings (bugs, vulnerabilities, code smells, security hotspots), backed by local SQLite, 4 MCP tools (`register_repo`, `analyse_repo`, `get_file_analysis`, `analyse_file`) + CLI for Claude Code hooks (SessionStart/PreToolUse/PostToolUse).

---

## 1. THE CRITICAL ARCHITECTURE DECISION

### 1.1 Option A — SonarQube Community Edition / Community Build Server

**Naming/versioning note (important, dated):** As of late 2024, "SonarQube Community Edition" was renamed **"SonarQube Community Build"**. Since version `24.12.0.100206`, the Community Build's release versioning diverged from the commercial "SonarQube Server" line and now follows calendar versioning `YY.M.0.BuildNumber` (e.g., `26.5`), released roughly monthly. Source: [endoflife.date/sonarqube-community](https://endoflife.date/sonarqube-community), [SonarQube Community Build release notes](https://docs.sonarsource.com/sonarqube-community-build/server-update-and-maintenance/release-notes), [SonarQube Server 2025 Release 1.8 LTA](https://community.sonarsource.com/t/sonarqube-server-2025-release-1-8-lta/182977).

**Docker setup & startup:**
- Official image: `docker.io/library/sonarqube` ([Docker Hub](https://hub.docker.com/_/sonarqube)).
- Requires `vm.max_map_count >= 524288` on Linux hosts (Elasticsearch dependency) and at least **4 GB RAM** allocated to the container (≈2GB app + Elasticsearch heap; 6-8GB recommended for larger projects). Source: [Server host requirements | SonarQube Community Build](https://docs.sonarsource.com/sonarqube-community-build/server-installation/server-host-requirements).
- First-boot startup is typically **60–120 seconds** (sometimes 2-3 minutes) while Elasticsearch indices initialize; subsequent restarts with persisted volumes are faster but still tens of seconds. Source: multiple setup guides incl. [How to Install SonarQube with Docker](https://dev.to/rahulxsingh/how-to-install-sonarqube-with-docker-659), [SonarQube Docker setup 2026 guide](https://aicodereview.cc/blog/sonarqube-docker/).

**REST API for results:**
- `/api/issues/search` supports filtering by rule, resolution, project, etc. The `types=BUG,VULNERABILITY,CODE_SMELL` parameter exists but **is deprecated** in newer Community Build versions (10.4.1+) in favor of other filters. Source: [community.sonarsource.com - api/issues/search types deprecated](https://community.sonarsource.com/t/api-issue-api-issues-search-types-is-deprecated/110588).
- Rule type taxonomy (BUG / VULNERABILITY / CODE_SMELL / SECURITY_HOTSPOT) is a core part of the data model — confirmed via [Security-related rules docs](https://docs.sonarsource.com/sonarqube-server/10.8/user-guide/rules/security-related-rules) and rule metadata JSON (`type`, `severity`, `tags`, `status` fields), catalog at [rules.sonarsource.com](https://rules.sonarsource.com/).

**Licensing / Community Build limitations** (source: [Is SonarQube Free? Community Edition Explained](https://dev.to/rahulxsingh/is-sonarqube-free-community-edition-explained-20l1), [SonarQube editions differences](https://www.almtoolbox.com/blog/sonarqube-editions-differences/), [Managing Security Hotspots | Community Build docs](https://docs.sonarsource.com/sonarqube-community-build/user-guide/security-hotspots)):
- No branch analysis (main branch only), no PR decoration, no taint-analysis-based vulnerability detection.
- **Security hotspots ARE present in Community Build's data model** but lack the dedicated accept/reject review workflow (Enterprise feature) — however, the underlying rule type and issue data are still exposed via the API, so an MCP tool *can* read them.
- C, C++, Objective-C, Swift, PL/SQL, ABAP, T-SQL, COBOL, RPG, Apex are paid-edition-only languages. **TypeScript/JavaScript and C# are fully supported in Community Build.**
- Compliance reports (OWASP/CWE/PCI-DSS mappings) require Enterprise.

**Single-file vs full-project scans:**
- `sonar-scanner` is designed around full-project analysis; the analysis report is built client-side then uploaded and processed **asynchronously** (queued) server-side. There is no first-class "analyze just this one file synchronously and get JSON back in milliseconds" API.
- You *can* scope a scan with `sonar.inclusions`/`sonar.exclusions` to a subset of files, but you still pay the cost of scanner JVM startup, project metadata bootstrap, and a queued server-side processing step. Community discussion confirms users struggle even to get full monolith scans under ~20 minutes, with single-line changes still taking substantial time end-to-end if done via full-project CI scans. Source: [SonarQube PR Analysis for huge monolithic applications](https://medium.com/nerd-for-tech/sonarqube-pull-request-analysis-for-huge-monolithic-applications-62533403e35f), [Sonar scanner for only required files](https://community.sonarsource.com/t/sonar-scanner-for-only-required-files-in-project/107636).
- **Verdict on speed:** scanner JVM startup alone (~5-15s) plus async server processing makes this fundamentally unsuitable for a synchronous sub-few-second PostToolUse hook on every file edit.

**New finding — `SonarSource/sonarqube-cli` ("sonar" CLI):**
- A genuinely new (2025/2026) official tool: [SonarSource/sonarqube-cli](https://github.com/SonarSource/sonarqube-cli), latest version **1.0.0.2628 (released June 10, 2026)**, GA, LGPL-3.0.
- Installed via `curl ... install.sh`; provides `sonar analyze --file <file>` which analyzes **uncommitted changes in a specific file** for new issues, plus secrets scanning (marked BETA) and AI-assisted remediation suggestions.
- **However**, it still requires connecting to **SonarQube Cloud (v9.9+) or a self-hosted Server** — it is a thin client over the same server-backed analysis pipeline, not a standalone local analyzer. This reinforces that anything carrying the "SonarQube"/"Sonar" brand name for full rule fidelity assumes a server backend (cloud or self-hosted).

**Official SonarQube MCP Server (directly relevant prior art — see Section 2):**
- [SonarSource/sonarqube-mcp-server](https://github.com/SonarSource/sonarqube-mcp-server) requires Docker/OCI runtime + a SonarQube Server or Cloud connection + token auth. Licensed under the **SONAR Source-Available License v1.0** (not OSI-approved open source; restricts "Competitive" use). It does have an `analyze_code_snippet` tool that can do **local analysis without server upload** for some languages — but the overall server still mandates a SonarQube backend connection for the issue/hotspot/quality-gate tools. Source: [docs.sonarsource.com/sonarqube-mcp-server](https://docs.sonarsource.com/sonarqube-mcp-server), [SonarQube MCP Server announcement](https://www.sonarsource.com/resources/library/model-context-protocol/).

### 1.2 Option B — Standalone analyzer libraries (no server)

#### TypeScript/JavaScript: `eslint-plugin-sonarjs`

- **Critical status finding:** The repo `SonarSource/eslint-plugin-sonarjs` was **archived by its owner on Oct 3, 2024**. That archived repo covers versions **up to 1.0.0** and only exposed a *subset* of SonarJS rules (37 rules: 10 "Bug" + 27 "Code Smell" categories — **no vulnerability/security-hotspot rules** in that legacy version). Source: [eslint-plugin-sonarjs README (archived)](https://github.com/SonarSource/eslint-plugin-sonarjs/blob/master/README.md).
- **Active development moved to `SonarSource/SonarJS`** monorepo. The npm package `eslint-plugin-sonarjs` is still published from there, now at **v4.0.3** (per npm registry query, `_npmVersion: 11.11.0`, `_nodeVersion: 24.14.1`, peer dep `eslint: ^8.0.0 || ^9.0.0 || ^10.0.0`, `typescript: >=5` required as a dependency). Source: live npm registry query (`registry.npmjs.org/eslint-plugin-sonarjs/latest`).
- Versions **≥2.0.0 expose ALL SonarJS rules** (not the old curated subset): per the SonarJS repo, current coverage is roughly **JavaScript: 475 rules, TypeScript: 492 rules, CSS: 30 rules** (≈997 total across languages, including JSX/Flow/Vue/AWS Lambda support). Source: [SonarSource/SonarJS](https://github.com/SonarSource/SonarJS) repo description, latest analyzer release **12.5.0.41048** (dated May 18, 2026).
- **Flat config**: v4.x fully supports ESLint 9 flat config (`eslint.config.js`) and provides a legacy `recommended-legacy` config for ESLint 8. Node.js ≥16 required. Source: README fetch above; also tracked in [Working with ESLint 9 issue #438](https://github.com/SonarSource/eslint-plugin-sonarjs/issues/438) and [eslint-plugin-sonarjs v2 changes thread](https://community.sonarsource.com/t/eslint-plugin-sonarjs-v2-changes/124011).
- **Rule categorization**: Each rule's metadata includes Sonar's `type` (Bug / Code Smell / Vulnerability / Security Hotspot), `severity`, `tags`, matching the same taxonomy as SonarQube's rule repository at [rules.sonarsource.com](https://rules.sonarsource.com/) (rule keys like `S1234`, `S2589`, etc.). The plugin generates this metadata via `scripts/generate-rules-sonar-meta.ts` in the SonarJS repo — meaning **yes, the npm package DOES carry genuine Sonar rule-key/type/severity classification**, not generic ESLint warnings. Source: [generate-rules-sonar-meta.ts](https://github.com/SonarSource/eslint-plugin-sonarjs/blob/master/scripts/generate-rules-sonar-meta.ts).
- **No separate `@sonar/*` programmatic-without-ESLint package exists currently.** The old standalone `sonarjs` npm package's CLI is explicitly deprecated; SonarSource's stated direction is "use SonarQube or eslint-plugin-sonarjs." Source: npm search results for `sonarjs`/`eslint-plugin-sonar`.
- **Run time**: ESLint with this plugin runs as a normal ESLint rule-set — single-file lints with ESLint are typically **sub-second to ~1-2 seconds** (dominated by ESLint/TS-parser startup, not the 992-rule count, since SonarJS rules reuse the shared TS AST). This is well within PostToolUse hook budgets. Full-repo runs scale roughly linearly and parallelize well via standard ESLint caching/workers — no fundamental architectural reason it can't run on a large repo in reasonable time (minutes, not the 20min-1.5hr figures seen for full SonarQube scans).

#### .NET/C#: `SonarAnalyzer.CSharp`

- This is the real, actively-shipped NuGet package: Roslyn analyzers that "spot Bugs, Vulnerabilities and Code Smells" — confirmed description from [NuGet Gallery](https://www.nuget.org/packages/SonarAnalyzer.CSharp/). Repo: [SonarSource/sonar-dotnet](https://github.com/sonarsource/sonar-dotnet).
- **Current version (verified live via NuGet API):** the package has an extremely active release cadence — version list pulled directly from `api.nuget.org` shows the latest published version is **`10.27.0.140913`**, with `10.26.0.140279` and `10.25.0.139117` etc. immediately preceding it. This confirms "healthy, near-weekly release cadence."
- **Integration**: standard `dotnet add package SonarAnalyzer.CSharp` with:
  ```xml
  <PackageReference Include="SonarAnalyzer.CSharp">
    <PrivateAssets>all</PrivateAssets>
    <IncludeAssets>runtime; build; native; contentfiles; analyzers</IncludeAssets>
  </PackageReference>
  ```
  Once referenced, analyzers run automatically as part of `dotnet build` / `dotnet restore`+`build`, surfacing results as standard MSBuild warnings/errors in build output. Rules can be tuned via a `.ruleset` file or `.editorconfig` severity entries (e.g., `dotnet_diagnostic.S1234.severity = warning`), matching Sonar rule keys (`S####`) directly — confirming rule-key-level classification is preserved end-to-end. Source: [NuGet Gallery SonarAnalyzer.CSharp 10.27.0.140913](https://www.nuget.org/packages/SonarAnalyzer.CSharp/), [Setting Up SonarLint/SonarAnalyzer.CSharp in .Net Core](https://community.sonarsource.com/t/setting-up-sonarlint-sonaranalyzer-csharp-in-net-core/10640).
- **Single file without full build**: As packaged, SonarAnalyzer.CSharp ships as Roslyn `DiagnosticAnalyzer`s that hook into the Roslyn `Compilation` pipeline — they fundamentally operate over a `Compilation` (which needs project/reference context for full semantic analysis), not isolated files. There is **no documented official standalone single-file CLI** for SonarAnalyzer.CSharp; research turned up no `AdhocWorkspace`/`GetAnalyzerResult` sample from SonarSource itself (only generic Roslyn analyzer-testing patterns from `dotnet/roslyn-analyzers` discussions). Practical options:
  1. `dotnet build` (or `dotnet build --no-restore` on a pre-restored project) — runs full analyzer pass for the project containing the changed file. Can be slow on first run but **incremental builds with `RunAnalyzers=true` are reasonably fast** (seconds) once the project is warm, per [Optimizing C# code analysis for quicker .NET compilation](https://anthonysimmon.com/optimizing-csharp-code-analysis-for-quicker-dotnet-compilation/).
  2. `dotnet format analyzers --severity info --verify-no-changes` — runs analyzers across a project/solution and reports diagnostics; still project-scoped, not single-file, and can be slow on large solutions.
  3. Roslyn `AdhocWorkspace`/`CSharpCompilation.Create()` hosting the analyzer DLL directly against a single `SyntaxTree` + minimal reference set — technically possible (this is how SonarLint's language server and IDE plugins do "single file" analysis) but **requires building a custom host**; not an out-of-the-box CLI.
- **SARIF support**: `dotnet build` supports `/p:ErrorLog=results.sarif,version=2.1` to emit SARIF 2.1 output containing rule IDs (`S####`), severity, and locations — SonarSource also publishes general guidance on SARIF interop. Source: [The complete guide to SARIF | Sonar](https://www.sonarsource.com/resources/library/sarif/), [SARIF reports | SonarQube Server docs](https://docs.sonarsource.com/sonarqube-server/analyzing-source-code/importing-external-issues/importing-issues-from-sarif-reports).
- **Rule metadata/classification**: rules carry Sonar keys (`S####`) with type (Bug/Vulnerability/Code Smell) — same taxonomy as the SonarQube rule repository, browsable at [rules.sonarsource.com](https://rules.sonarsource.com/csharp/).

### 1.3 RECOMMENDATION: Option B (standalone analyzers), hybrid per-language

**Decision: Build on Option B — standalone analyzer libraries — for BOTH ecosystems. Do NOT make a SonarQube server/Docker a hard dependency.**

Evaluated against the four criteria:

| Criterion | Option A (SonarQube CE/Community Build server) | Option B (standalone analyzers) |
|---|---|---|
| (a) Parallel whole-repo analysis reasonably fast | Poor — async server processing, queued reports, JVM scanner overhead; community reports 20min-1.5hr on large monoliths | Good — ESLint (with caching/parallel workers) and `dotnet build` with analyzers are designed for repo-scale CI use; minutes not hours for typical repos |
| (b) Single-file sync analysis sub-few-seconds for PostToolUse | Fails — no sync single-file API; scanner+server round trip is seconds-to-minutes, async | TS/JS: yes, ESLint single-file lint is sub-second to ~1-2s. C#: borderline — needs a warm/incremental `dotnet build` of the containing project (seconds when warm); true sub-second single-file needs a custom Roslyn host (Phase 2 stretch) |
| (c) No mandatory external server/Docker | Fails by definition | Pass — both `eslint-plugin-sonarjs` (npm dep) and `SonarAnalyzer.CSharp` (NuGet analyzer package, runs in-process via Roslyn) need zero servers |
| (d) Genuine Sonar bug/vuln/code-smell/hotspot classification | Yes, but Community Build security-hotspot *workflow* is Enterprise-gated (data still present) | Yes for both: `eslint-plugin-sonarjs` v4.0.3 (≈992 rules across JS/TS/CSS, SonarJS analyzer 12.5.0.41048) carries Sonar rule-type/severity/tag metadata generated from the same source as SonarQube; `SonarAnalyzer.CSharp` 10.27.0.140913 rules are literally `S####` Sonar rule keys with Bug/Vulnerability/Code-Smell types |

**Concrete architecture recommendation:**

- **TS/JS**: Run ESLint programmatically (`ESLint` class from the `eslint` package, not the CLI) with `eslint-plugin-sonarjs` v4.0.3 loaded via flat config. This gives per-file results in well under a second for incremental hook use, and can run in parallel across the repo for `analyse_repo`. Map each finding's rule metadata (`type`, Sonar rule key, severity, tags) directly into your SQLite schema's bug/vulnerability/code-smell/security-hotspot columns.
- **C#**: Use `dotnet build` (or `dotnet build --no-restore -consoleloggerparameters:NoSummary` with `/p:ErrorLog=...sarif`) on the containing project, parsing SARIF output for `S####` rule diagnostics from `SonarAnalyzer.CSharp` 10.27.0.140913. For `analyse_repo`, run once per project/solution (parallelizable across projects). For `analyse_file`/PostToolUse, accept that first-run cost is higher than TS/JS but MSBuild incremental builds with analyzers warm-cached are typically a few seconds — acceptable for an async-reported PostToolUse hook (don't block the agent; report findings on next turn via `additionalContext`). A custom lightweight Roslyn host (`AdhocWorkspace`) for true single-file sub-second checks is a legitimate Phase 2 enhancement but is NOT how SonarSource ships the analyzer out of the box.
- **Explicitly do not bundle/require Docker or a SonarQube server.** This is a major differentiator vs. the official `SonarSource/sonarqube-mcp-server` (which mandates Docker + SonarQube Cloud/Server + a non-OSI license). Position your product as "Sonar-grade rule classification, zero infrastructure."
- If users *already* run SonarQube (self-hosted or Cloud), consider an optional/future "bring your own SonarQube" mode that supplements local findings with server-side data (taint analysis, hotspot review workflow) — but this should be additive, not foundational.

This is the product's white space.

## 3. TECHNICAL APPROACHES FOR REMAINING COMPONENTS

### 3.1 SQLite in Node.js: `better-sqlite3` vs `node:sqlite` vs `sql.js`

| | `better-sqlite3` | `node:sqlite` (built-in) | `sql.js` |
|---|---|---|---|
| Latest version (verified via npm registry) | **12.10.1** | N/A (built into Node runtime) | **1.14.1** |
| Node compatibility | `engines: "20.x \|\| 22.x \|\| 23.x \|\| 24.x \|\| 25.x \|\| 26.x"` (per live npm registry data) | Requires **Node ≥ 22.5** (experimental/active development, stability 1.1 "Active Development" per [Node 22 docs](https://nodejs.org/docs/latest-v22.x/api/sqlite.html)); promoted further in Node 24 | Pure WASM/asm.js, runs anywhere incl. browsers |
| API style | Synchronous, native bindings (prebuilt binaries via `prebuild-install`, falls back to `node-gyp`) | Synchronous (`DatabaseSync`, `StatementSync`) — explicitly synchronous-only per Node docs | Synchronous but in-memory (no native file I/O without extra glue; persists via export/import of byte arrays) |
| Fit for CLI + MCP server + hook scripts | **Best fit** — mature (years of production use, e.g. powers many Electron/CLI apps), synchronous API ideal for short-lived hook scripts that need to read/write the DB and exit quickly, no event-loop concerns | Good, but "Active Development" stability label means API can still change between Node versions; requires pinning Node ≥22.5/24, which may be too new a constraint for some users' environments in 2026 | Poor fit — WASM overhead and in-memory model add complexity for a file-backed persistent DB; better suited to browser/sandbox contexts |

**Recommendation**: Use **`better-sqlite3` (v12.10.1)** as the primary dependency for the MVP — broadest Node version support, synchronous API matches the "hook script does a quick DB read/write and exits" usage pattern perfectly, and is the de facto standard for this exact use case. Optionally add a thin abstraction layer so `node:sqlite` could be swapped in later once its API stabilizes across LTS lines (Node 22 LTS and 24 both ship it, but "Active Development" status means breaking changes are still possible). Sources: live `registry.npmjs.org` queries for `better-sqlite3@latest` and `sql.js@latest`; [Node.js v22 SQLite docs](https://nodejs.org/docs/latest-v22.x/api/sqlite.html).

### 3.2 Dependency/import graph extraction — TypeScript

| Tool | Latest version (npm registry) | Node engine | Best for |
|---|---|---|---|
| `dependency-cruiser` | **17.4.3** | `^20.12 \|\| ^22 \|\| >=24` | Whole-repo dependency graphs with rich rule validation, JSON output (`--output-type json`), handles TS/JS/CommonJS/ESM; designed for CI use, has caching |
| `madge` | **8.0.0** | `>=18` | Quick CLI dependency trees, circular-dependency detection; simpler output, less configurable than dependency-cruiser |
| `ts-morph` | **28.0.0** | (no explicit `engines` field in package.json) | Programmatic TS Compiler API wrapper — not a turnkey dependency-graph tool, but gives full AST/type access; would require writing custom traversal of `SourceFile.getImportDeclarations()` / `getReferencedSourceFiles()` |
| TypeScript Compiler API directly | (part of `typescript` package) | — | Most control, most code to write; `ts.createProgram()` + `program.getSourceFiles()` + resolving `ts.resolveModuleName` gives exact per-file import lists including type-only imports |

**Recommendation**: For **per-file import list extraction** at the granularity needed for a SQLite dependency table, **`dependency-cruiser` (17.4.3)** is the most practical turnkey choice — it already produces a JSON graph of `{source file -> [resolved dependency files]}` with module-resolution baked in (respects `tsconfig.json` paths, node_modules resolution, etc.), avoiding the need to hand-roll TS Compiler API traversal. `ts-morph` (28.0.0) is the right tool only if you also need deeper AST manipulation/refactoring beyond dependency extraction (it would be overkill just for import graphs). `madge` (8.0.0) is a lighter-weight fallback if dependency-cruiser's output proves too heavy. Sources: live npm registry queries for `dependency-cruiser@latest`, `ts-morph@latest`, `madge@latest`; [dependency-cruiser npm](https://www.npmjs.com/package/dependency-cruiser), [ts-morph.com](https://ts-morph.com/), [madge GitHub](https://github.com/pahen/madge).

### 3.3 Dependency/import graph extraction — C#/.NET

Three viable approaches, in increasing order of fidelity/complexity (source: [Using the Roslyn APIs to Analyse a .NET Solution](https://www.stevejgordon.co.uk/using-the-roslyn-apis-to-analyse-a-dotnet-solution), [Get started with syntax analysis (Roslyn APIs)](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/get-started/syntax-analysis), [improveandrepeat.com - Access C# Projects with Roslyn](https://improveandrepeat.com/2023/05/access-your-c-projects-through-code-with-roslyn/)):

1. **`.csproj` `<ProjectReference>`/`<PackageReference>` parsing** — cheapest, gives project-level (not file-level) dependency edges. Good for a coarse "which projects depend on which" graph but not per-file imports.
2. **Syntax-tree `using` directive collection** — use `Microsoft.CodeAnalysis.CSharp` to parse each `.cs` file's `SyntaxTree`, walk for `UsingDirectiveSyntax` nodes. Cheap (no full compilation needed), gives per-file namespace-level dependencies — closest C# analog to a TS "import list," though it's namespace-based rather than file-based (C# doesn't have direct file-to-file imports the way ES modules do).
3. **Full `MSBuildWorkspace` + `Compilation` + symbol resolution** — load the `.sln`/`.csproj` via `MSBuildWorkspace.OpenProjectAsync`, get a `Compilation`, and use semantic model (`GetSymbolInfo`) to resolve which *files* (via `SyntaxReference.SyntaxTree.FilePath`) actually define the types referenced by `using`s — gives true per-file dependency edges but requires a working full build environment (NuGet restore, SDK resolution) and is significantly heavier/slower than (2). Tools like the community "RoslynCodeAnalyzer" graph generators (e.g. [devfire/RoslynCodeAnalyzer](https://github.com/devfire/RoslynCodeAnalyzer)) follow this pattern, outputting nodes/edges as JSON.

**Recommendation**: Start with **approach 2 (syntax-only `using` directive extraction via `Microsoft.CodeAnalysis.CSharp.CSharpSyntaxTree.ParseText`)** for the MVP dependency graph — it's fast (no MSBuild/restore needed), works file-by-file (good for incremental hook updates), and is "good enough" for surfacing "files that reference namespace X." Reserve full `MSBuildWorkspace`-based symbol resolution (approach 3) for a later "precise impact analysis" feature, since it requires a restored build environment and is much slower.

### 3.4 `@modelcontextprotocol/sdk` (TypeScript)

- **Current version (verified via npm registry): `1.29.0`**, published by `Anthropic, PBC`, MIT license, `"type": "module"`, `engines: { "node": ">=18" }`.
- Implements stdio and Streamable HTTP transports; stdio is the relevant transport for a locally-spawned CLI/MCP server pattern (Claude Code spawns it as a child process). Basic pattern: instantiate `McpServer`, register tools via `server.tool(name, schema, handler)` (Zod schemas for input validation — peer dependency on `zod`, with internal usage of `zod/v4` while maintaining backward compatibility with `zod ^3.25+`), create `StdioServerTransport`, call `server.connect(transport)`. Source: live npm registry query for `@modelcontextprotocol/sdk@latest`; [typescript-sdk GitHub](https://github.com/modelcontextprotocol/typescript-sdk), [typescript-sdk/docs/server.md](https://github.com/modelcontextprotocol/typescript-sdk/blob/main/docs/server.md).

### 3.5 Claude Code Hooks (verified against official docs at `code.claude.com/docs/en/hooks`, redirected from `docs.claude.com`)

- **SessionStart**: fires on session start/resume/clear/compact (matcher values: `startup`, `resume`, `clear`, `compact`). Hook receives JSON via stdin with `session_id`, `transcript_path`, `cwd`, `hook_event_name: "SessionStart"`, `source`, `model`. Can return `additionalContext`, `sessionTitle`, `initialUserMessage`, `watchPaths` via `hookSpecificOutput` — cannot block. **Use case for this product**: on session start, call `register_repo`/`analyse_repo` (or check if already registered) and inject a summary of current findings as `additionalContext`.
- **PreToolUse**: fires before a tool executes; can block via `permissionDecision: "deny"|"allow"|"ask"|"defer"` in `hookSpecificOutput`. Receives `tool_name` and `tool_input` (e.g., for `Edit`/`Write`, `tool_input.file_path` is present). Matchers are exact tool names or regex (`Edit|Write`, `mcp__.*`); an additional `if` field supports permission-rule syntax like `Edit(*.ts)`. **Use case**: not essential for this product's core loop, but could be used to pre-emptively warn the agent about known issues in a file it's about to edit.
- **PostToolUse**: fires after a tool succeeds; cannot block but can return `additionalContext`/`updatedToolOutput`, or exit code 2 to surface stderr feedback to Claude. Receives `tool_name`, `tool_input` (incl. `file_path`), and `tool_response`. **Use case (primary integration point)**: matcher `Edit|Write` → invoke `analyse_file` CLI command on the just-edited file, write results to SQLite, and return findings as `additionalContext` so Claude immediately sees new bugs/vulnerabilities/code-smells introduced by its own edit.
- Config lives in `settings.json` under `"hooks": { "<EventName>": [ { "matcher": "...", "hooks": [ { "type": "command", "command": "...", "timeout": <seconds> } ] } ] }`. `${CLAUDE_PROJECT_DIR}` is available for path interpolation.

Source: [code.claude.com/docs/en/hooks](https://code.claude.com/docs/en/hooks) (redirected from `docs.claude.com/en/docs/claude-code/hooks`).

## Recommendation Summary

Build `mcp-sonar-analysis` on **Option B (standalone, server-free analyzers)**: for TypeScript/JavaScript, run **`eslint-plugin-sonarjs` v4.0.3** (≈992 Sonar-classified rules across JS/TS/CSS, flat-config compatible, ESLint ≥8) programmatically via the `eslint` package's `ESLint` class — this is fast enough (sub-1-2s per file) for a synchronous PostToolUse hook and scales to whole-repo `analyse_repo` runs via standard ESLint parallelism/caching; for C#/.NET, run **`SonarAnalyzer.CSharp` v10.27.0.140913** as Roslyn analyzers via `dotnet build` with `/p:ErrorLog=...sarif` (parse SARIF for `S####` rule diagnostics), accepting that single-file C# checks ride along with incremental project builds (seconds when warm) rather than true sub-second isolation — a custom Roslyn `AdhocWorkspace` host is a viable Phase 2 upgrade but not required for MVP. **Do not require Docker or a SonarQube server** — this is the single biggest differentiator versus SonarSource's own official MCP server (`SonarSource/sonarqube-mcp-server`, Docker + SonarQube Cloud/Server + Source-Available License v1.0) and most "sonarqube mcp" community projects. Persist all findings plus a per-file dependency graph in SQLite via **`better-sqlite3` v12.10.1** (broadest Node compatibility, synchronous API ideal for short-lived hook processes); extract TS/JS import graphs with **`dependency-cruiser` v17.4.3** and C# dependency edges via lightweight Roslyn syntax-tree `using`-directive parsing (no MSBuild restore needed for MVP). Build the MCP layer on **`@modelcontextprotocol/sdk` v1.29.0** (stdio transport, Zod schemas) and wire the four tools (`register_repo`, `analyse_repo`, `get_file_analysis`, `analyse_file`) into Claude Code's **SessionStart** (bootstrap/summary via `additionalContext`), and **PostToolUse** on `Edit|Write` matchers (run `analyse_file`, feed fresh findings back as `additionalContext`) per the hook schema documented at `code.claude.com/docs/en/hooks`. This positions the product in the validated-but-unfilled gap: Sonar-grade bug/vulnerability/code-smell/security-hotspot classification, zero mandatory infrastructure, cross-language (TS/JS + C#) unification, local SQLite-backed dependency graph, and proactive hook-driven surfacing — none of which any single competitor (official SonarQube MCP, sonarqube-cli, Semgrep MCP, ESLint MCP, or the codegraph-style local-intelligence MCP servers) currently combines.
